# Task Manager — NextJS 15 + tRPC

A simple task management application, built as a technical challenge.
Frontend and backend live in a single NextJS project (App Router), with the
backend exposed via **tRPC** and data kept **in memory** (no database).

## Stack

- **Next.js 15** (App Router, Server Components)
- **tRPC v11** (type-safe backend, `fetch` adapter)
- **React Query v5** (async state/cache on the client, via `@trpc/react-query`)
- **Zod** (input validation, shared between client and server)
- **TypeScript** (end-to-end typing: the frontend automatically infers the backend's types)
- Plain CSS (no framework), to keep the project simple

## How to run

Prerequisite: Node.js 18.18+ (20+ recommended).

```bash
npm install
npm run dev
```

Open [http://localhost:3000](http://localhost:3000).

To run in production mode:

```bash
npm run build
npm run start
```

No environment variables or database setup are required — data lives in
memory in the Node.js server process, and the app already ships with 3
sample tasks (purely fictional) to make testing easier.

## Project structure

```
src/
  server/
    db.ts                 # in-memory "database" (array of tasks)
    trpc.ts                # tRPC initialization (context, error formatter)
    routers/
      task.ts              # task CRUD (create/list/getById/update/delete)
      _app.ts               # root router (appRouter) + AppRouter type
  trpc/
    client.ts              # typed hooks (createTRPCReact<AppRouter>)
    Provider.tsx            # QueryClientProvider + trpc.Provider (client)
    server.ts                # caller used in Server Components (SSR)
  app/
    layout.tsx              # root layout
    globals.css              # styles
    page.tsx                  # task listing page (SSR)
    tasks/
      new/page.tsx            # task creation
      [id]/edit/page.tsx        # task editing (SSR + 404 if not found)
    api/trpc/[trpc]/route.ts    # HTTP endpoint that exposes appRouter
  components/
    TaskList.tsx             # list (client): delete + infinite scroll
    TaskForm.tsx              # reusable form (create/edit)
```

## Key technical decisions

- **Real SSR on the listing page**: `app/page.tsx` is a Server Component
  that calls `appRouter` directly on the server (via `createServerCaller`),
  without going through HTTP. This guarantees the first page of tasks is
  already ready in the initial HTML. On the client, `useInfiniteQuery` is
  hydrated with that same result (`initialData`), avoiding a duplicate
  fetch when the page mounts.

- **Infinite scroll (bonus)**: the `task.list` endpoint accepts
  `cursor`/`limit` and returns `nextCursor`. On the client, a "sentinel"
  element is observed with `IntersectionObserver`; when it enters the
  viewport, the next page is fetched automatically (`fetchNextPage`).

- **Two-layer validation**: the form validates the title on the client
  (instant feedback), and the backend validates again with Zod — the
  server is always the source of truth. Zod validation errors are passed
  through in a structured way (`fieldErrors`) via tRPC's `errorFormatter`,
  allowing the specific form field's error message to be displayed.

- **Business error handling**: trying to update/delete a task with a
  nonexistent `id` returns a `TRPCError({ code: "NOT_FOUND" })`, with a
  clear message — handled both on the edit page (via Next.js's
  `notFound()`) and in the client mutations (error message shown to the user).

- **UI feedback**: deleting a task shows a loading state
  ("Deleting..."), a success/error message after the operation, and a
  simple `confirm()` before removing it.

- **In-memory data with `globalThis`**: to avoid losing state on every
  Next.js hot-reload in development, the task array is stored on
  `globalThis`. This is just a dev convenience — the data is still lost on
  every process *restart*, as expected for an in-memory "database".

## Known limitations (expected for this scope)

- No real persistence: restarting the server clears all tasks.
- No authentication/multi-user support.
- In serverless environments with multiple instances, each instance would
  have its own in-memory state — in production, this would be replaced by
  a real database (Postgres, SQLite, etc.).

## Part 1 — Theoretical Answers

### 1. ReactJS and NextJS: Rendering and Strategies

**a) CSR vs SSR vs SSG**

- **CSR (Client-Side Rendering)**: the server sends a mostly empty HTML
  shell + the JS bundle; React mounts the UI and fetches the data in the
  user's browser. This is the classic "SPA" model.
- **SSR (Server-Side Rendering)**: the HTML is already generated by the
  server, on every request, with the data already loaded. The user sees
  content faster and the HTML is indexable by search engines, but every
  request generates work on the server.
- **SSG (Static Site Generation)**: the HTML is generated **at build
  time**, before any request happens, and served as a static file (or via
  a CDN). It's the fastest of the three, but the content only changes with
  a new build (or revalidation, in the case of ISR).

**b) When each one is advantageous/problematic**

- **CSR** is advantageous for internal dashboards, authenticated areas,
  and highly interactive tools where SEO doesn't matter — but it's bad for
  content that needs to be indexed or load fast on the first visit (the
  user sees a blank screen/loading state until the JS loads and fetches
  the data).
- **SSR** is ideal for pages with frequently changing data that also need
  good SEO/first paint, such as a feed, a product page with real-time
  stock, or — as in this challenge — a task list. It's problematic when
  traffic volume is very high, since every visit generates work on the
  server (more cost, higher latency if the backend/DB is slow).
- **SSG** is perfect for content that rarely changes: blogs,
  documentation, landing pages, institutional pages. It's problematic for
  highly dynamic or per-user personalized data, since it would require
  constant rebuilds (Next.js partially solves this with ISR — incremental
  regeneration — but it still isn't "real time").

### 2. Backend: Concepts and Approaches

**a) Main ways to implement a backend for a web application**

- **REST**: resource-based HTTP API (`GET /tasks`, `POST /tasks`), widely
  adopted, easy to cache and document (OpenAPI/Swagger).
- **GraphQL**: a single endpoint where the client specifies exactly which
  data it wants; reduces over-fetching/under-fetching, but adds setup
  complexity (schema, resolvers) and caching complexity.
- **RPC-style (e.g. tRPC, gRPC)**: the client calls backend "functions"
  almost as if they were local. tRPC specifically uses TypeScript to share
  types between client and server without needing a separate schema (like
  GraphQL) or code generation.
- **BaaS / serverless functions**: isolated functions (e.g. AWS Lambda,
  Vercel Functions) per endpoint, without a traditional continuously
  running "server" — good for variable workloads and small teams.
- **Traditional monolith** (Express, Fastify, NestJS, Django, Rails,
  etc.): a single server handling routes, business logic, and data access.

**b) tRPC — advantages and disadvantages**

Advantages:
- **Automatic end-to-end typing**: the `appRouter` type on the backend is
  imported on the frontend, so any change to the API immediately breaks
  the client build (a development-time error, not a production one).
- **No code generation or separate schema**: unlike GraphQL, there's no
  need for `.graphql` files, codegen, or manually keeping a schema in
  sync — it's just TypeScript.
- **Built-in validation**: combined with Zod, each procedure already
  validates its input and automatically infers the type from the schema.
- **Productivity**: for full-stack teams working in TypeScript, it's
  extremely fast to develop with — which is exactly why it was used in
  this challenge.

Disadvantages:
- **Coupling to TypeScript/monorepo**: it only works well when client and
  server are in the same repository (or share the types package). It's
  not suitable if you need to expose the API to external consumers who
  don't use TypeScript (native mobile teams, external partners, etc.) —
  in those cases REST or GraphQL are more interoperable.
- **Security**: tRPC by itself doesn't solve authentication/authorization
  — that needs to be implemented in the `context` and in procedure
  middlewares, just like in any other backend. Care must also be taken not
  to leak sensitive information in errors (this project's
  `errorFormatter`, for example, only exposes a structured `zodError`, not
  the full stack trace).
- **Scalability**: technically it scales like any other HTTP API (tRPC
  runs over HTTP under the hood), but "scalability" here is more about
  code maintenance: in very large teams or with multiple heterogeneous
  consumers, the tight client/server coupling can become an organizational
  disadvantage, even though it's a technical advantage in smaller, more
  cohesive teams.
